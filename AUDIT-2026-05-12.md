# xiao-t-arena Codebase Audit
**Date:** 2026-05-12
**Goal:** 評估把現有 codebase 重構成「可重複交付給其他英文老師」的工作量、找出哪些東西要抽、哪些可以砍、Firebase 要怎麼複製。

---

## 📌 已敲定的策略決定（2026-05-12 對話結論）

- **首要買家**：英文獨立教育老師（NGFA-side），不是中文兒財。
- **客戶技術力**：完全不會 code → 你做的是 service（setup + hosted），不是 self-serve 模板。
- **行銷敘事（A+C 混合）**：**"A reflective learning platform for kids — where teachers actually read what your child writes."** 定位 = 反 ClassDojo 的膚淺 + 反演算法 app 的噪音。Pinterest / NGFA 對外的內容方向都用這條主軸。
- **影片模式**：平台只放 link（embed YouTube/Vimeo），不自架。中文 Tiffany 兒財繼續用 email 寄連結，NGFA 開 embed。Phase 1 多 2-4 hr 加 `videoUrl` 欄位。
- **Phase 1 範圍**：雙 brand config 重構（中文 + NGFA），不做完整 multi-tenant 抽象。第 3 個外部客戶來才開始抽。
- **🆕 交付模式：B (Admin UI 內容編輯器)**，2026-05-12 確認。客戶不寄內容給 Tiffany 編 JSON；客戶登入後台自己編。Tiffany 只做 tech setup + hosting，不碰內容。**架構含意：weekly content 從 hardcoded JS 移到 Firestore `brandContent` collection，admin UI 讀寫 Firestore，學員端從 Firestore 讀（TanStack Query 快取）**。
- **🆕 Phase 1 4 個 milestones**：(M1) 地基 + content → Firestore，~1-2 週；(M2) Admin content editor v1，~2-3 週；(M3) i18n + theme editor + NGFA 上線，~2-3 週；(M4) 對外賣 polish，等第一個外部客戶來才做。每個 milestone 都是 shippable state。

## TL;DR（一頁版）

1. **codebase 比預期乾淨**——你描述的 5 大功能（每週解鎖、三關制、兌換碼、後台審核、雙向對話、學習護照）**全部都在這個 repo**，不是分散在多個專案。Firebase 那層完全是 client + Firestore，沒有 server，這對「複製給別人」是好事。

2. **但「品牌耦合」程度非常高**——不是 logo/顏色那種表面耦合，是 **6 週的中文課程內容（含 60 題選擇題、開放題、任務文案、徽章/卡片名稱、鼓勵語）全部寫死在 source code 裡**。Source code 也有約 30-40% 的 dead code（從 base44 export 後沒清掉的舊頁面）。

3. **改成「雙 brand config」(中文兒財 + 英文 NGFA) 的 Phase 1 工作量：估 80-120 小時 / 4-6 週**（你每週 20 小時的話）。最大頭是 **內容抽離 + i18n + Firebase 多 project 切換**，不是寫新功能。

4. **三個 critical issues 要立刻處理**——和對外賣無關，是現有風險：
   - `migrationData.js` 把 30+ 個學員真實 email 寫死在 source code 裡（PII 外洩風險，git 已追蹤），應該移到 Firestore 或 env。
   - `firebase.js` 的 config 整段 hardcoded（雖然這些不是 secret，但這個 pattern 阻止你做雙 project）。
   - Firestore security rules 不在 repo（在 console），代表新 brand 要重新手動配。

5. **honest take**：我建議 **Phase 1 只做「NGFA 版能跑起來」，不做完整 multi-brand 抽象**。理由：你目前只需要 2 個 brand instance，過度抽象會讓 codebase 變難教（傷 Track B 教學包價值）。等真的有第 3 個客戶才開始抽。

## 3. Firestore 資料模型 + 多 project 複製計畫

### 3.1 現有資料模型

**`users` collection**（doc id = Firebase Auth UID）
```
{
  uid, email, displayName, photoURL,
  createdAt,
  weeklyProgress: { [week]: { score, total, completed, completedAt } },  // 舊欄位
  collection: { [rewardId]: { unlockedAt, codeUsed, source } },           // 主要使用
  migrated: boolean
}
```

**`weeklyProgress` collection**（doc id = `{userId}_week-{weekNumber}`）
```
{
  userId, weekNumber,
  // 練習題
  quizCompleted, quizCompletedAt, openAnswerContent, openAnswerSubmittedAt,
  openAnswerSeenAt, encouragementMessage, badgeEarned, badgeEarnedAt,
  // 任務
  taskText, taskImageUrls[], taskSubmittedAt, taskFeedback, taskCardCode,
  taskApprovedAt, taskRevealed, cardEarned, cardEarnedAt,
  // 對話
  conversation: [{ role: 'teacher'|'child', content, timestamp }],
  hasUnreadChildReply,
  createdAt, updatedAt
}
```

**`redeemCodes` collection**（doc id = 兌換碼字串）
```
{ uses: number, code: string }
```

### 3.2 「複製一套給新老師」的最少動作

對「完全不會 code」的客戶，是你幫他做，不是他自己做。流程：

1. **建新 Firebase project**（5 分鐘）：去 Firebase Console → 新增 project → 開 Firestore (asia-east1) + Storage + Auth (Google + Email/Password)。
2. **拿 6 個 config 值**：apiKey、authDomain、projectId、storageBucket、messagingSenderId、appId。
3. **設 Firestore Security Rules**：⚠️ 目前不在 repo，你 Phase 1 必做的事情之一是 **把現有 rules 從 console 抓出來、放進 repo**（建議 `firestore.rules` 檔 + `firebase.json`），未來 deploy 一行 `firebase deploy --only firestore:rules` 就能套用。
4. **設 Firestore Indexes**：你的程式碼有些 `where(...) where(...)` 複合查詢（例如 `getPendingBadges`），會需要複合索引。**這些也不在 repo**，要抓出來。
5. **設 Authorized domains**：把客戶的 production domain 加進 Firebase Auth 允許清單。
6. **clone repo + 改 `.env`**：repo 從 `firebaseConfig` 寫死改成 `import.meta.env.VITE_FIREBASE_*`，新客戶就只要改 `.env`。
7. **替換 brand config + 圖片**：根據 §2.B 抽出的 config 檔。
8. **vercel deploy**：給客戶 GitHub repo 權限 + 連 vercel project。

**每次交付一套 = 30-60 分鐘**（前提是 Phase 1 重構做完）。**這是你的單位經濟學基礎**。

## 5. 🚨 立刻處理的三個 critical issues

這三個和對外賣產品無關，是 **現有 production 的 latent risks**：

### 5.1 `src/lib/migrationData.js`：30+ 真實用戶 email 在 git history

```js
"shaneboy0119@gmail.com": { badges: [...], cards: [...] },
"tobyboy0214@gmail.com": { badges: [...], cards: [...] },
...
```

這是 base44 遷移時為了「老用戶登入後自動補徽章」做的對照表，但：
- 真實 email 寫在 source code、push 到 git、deploy 到 client bundle
- 任何拿到 production JS bundle 的人都能 grep 出這些 email
- 已經 commit 進 git history，即使現在改也擦不掉舊版本

**建議：**
1. 把這份對照表搬到 Firestore（一次性 admin script），AuthContext 改成從 Firestore 讀。
2. 從現在這個 commit 開始拿掉（雖然 git history 還有，但至少新版本乾淨）。
3. 如果你要更徹底，可以做 `git filter-repo` 清掉歷史——但會 break 任何 clone（包含 Vercel deploy 紀錄）。

### 5.2 Firestore Security Rules 不在 repo

如果你今天不小心在 Firebase Console 改錯一條 rule，沒辦法 rollback。對「複製給新老師」也是個障礙（沒有可複製的 rules artifact）。

**建議：**
1. 在 repo 加 `firebase.json` + `firestore.rules` + `firestore.indexes.json`
2. `npm install -g firebase-tools` → `firebase init firestore`（選現有 project）→ 它會把現有 rules 抓下來

### 5.3 Firebase API key 寫死在 `firebase.js`

```js
const firebaseConfig = {
  apiKey: "AIzaSyClH_4vqDUDZ5seyceBqRujbziXiIXOvYk",
  ...
};
```

**這不是 secret**（Firebase web config 設計上就是 public，安全靠 Security Rules，不是靠 key 保密）。但這個 pattern 阻止你做雙 project。改成 env 是 Phase 1 一定要做的。

## 7. 給 Track A、B、C 的具體建議（基於 audit 結果）

### Track A（賣成品 / 託管服務）的可行性
- ✅ 技術上完全可行——資料模型乾淨、邏輯通用、Firebase 模式好複製。
- ⚠️ 但客戶是「完全不會 code 的英文老師」，意思是 **你做的是 service 業務不是 product**。每接一個客戶 = 你花 30-60 分鐘 deploy + 後續 ongoing 維護負擔。
- 💡 建議定價結構：**setup fee USD $1,500-3,000（一次性）+ hosting USD $30-80/月**。Firebase Spark plan 對小流量免費，Blaze plan 超量再收，所以你 hosting 邊際成本接近零，月費接近純利潤。**但** 你會被綁住——客戶有問題會 ping 你。**接到第 5 個客戶之前你要建支援流程，不然就會壓垮你**。

### Track B（教學包）的可行性
- ✅ 你的 codebase 是個 **excellent 教學案例**——它 demo 了「用 Firebase 蓋一個真實在賣錢的訂閱式學習平台」的完整 stack。
- 💡 教學定位建議：「**How a Solo Educator Built Her Own Subscription Platform with AI + Firebase（no engineering background）**」——這個切角你有獨家信譽。
- 💡 教學內容建議：用你 Phase 1 重構過程當教材的素材（你已經在做了，順便錄）。整個重構 = 一個 6-8 週的 capstone 案例。
- 💡 定價結構：自學課 USD $200-500（影片+repo+Discord 社群），1-on-1 coaching add-on USD $500-1,500（4 次/2 週 sprint 帶你 build 出 MVP）。

### Track C（時間配比）
**Phase 1 期間（接下來 6-8 週）：100% 投在重構自己用** + **0% 對外**。
